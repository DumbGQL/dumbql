Complete guide for migrating an existing Apollo Client project to DumbQL. Use the CLI scanner to find all Apollo patterns, then follow the manual migration steps. For incremental migration, use the adapter layer.

Why Migrate?

Apollo Client is the most popular GraphQL client, but it has significant drawbacks for Angular projects: it's a React library ported to Angular, it requires complex typePolicies for cache normalization, and many features (file uploads, offline queue, auth refresh) require third-party packages. DumbQL solves all of this with a framework-native, modular architecture.

What you gain

  • Angular-native architecture — Signals, standalone DI, pipes, SSR with TransferState
  • Zero-config cache — auto __typename + id normalization, no typePolicies
  • Built-in middleware — auth refresh, retry, offline queue, APQ, batching, auto mock
  • Smaller bundle — ~3.2 kB core vs ~50 kB Apollo + you only import what you use
  • DevTools in-app — no browser extension required

What you lose

  • Ecosystem size — fewer community packages and tutorials (Apollo is more established)
  • Familiarity — team will need to learn new API patterns
  • Relay-style data masking — DumbQL doesn't have a compiler-based masking system

CLI Usage

The @dumbql/apollo-adapter package ships a CLI scanner that analyzes your project for Apollo patterns and generates a migration report. 

# Scan project for Apollo patterns
npx @dumbql/apollo-adapter src/

# Scan and auto-fix import paths (--fix)
npx @dumbql/apollo-adapter src/ --fix

# Scan a specific directory
npx @dumbql/apollo-adapter projects/my-app/src --fix

The scanner detects Apollo imports, API calls, and configuration patterns, then maps each one to the DumbQL equivalent. The --fix flag auto-replaces import paths where safe (e.g., @apollo/client → @dumbql/client, ApolloClient → DumbqlClient). Complex patterns (cache operations, Link chains) are flagged for manual migration.

Adapter: Incremental Migration

If you can't migrate your entire project at once, use the Apollo adapter to run both clients side by side. This is useful for large codebases where you want to migrate module by module.

The fromApolloCache() function wraps an Apollo InMemoryCache instance so it can be used as a DumbQL CacheStore. This lets you gradually migrate components one by one while keeping the Apollo cache as the source of truth.
